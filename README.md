# Suhail Airlines Insights

A standalone demo analytics dashboard built for a Pulses.ai internship stage-gate review.
It shows what a production computer-vision people-counting product could look like for an
airport terminal client — footfall, zone traffic, passenger "mood" scoring, and journey
flows — **entirely on generated mock data**. It does not connect to any camera, sensor, or
real dataset.

## Why it exists

Pulses.ai builds edge AI / computer-vision analytics. This project reproduces the shape of
that kind of product (the same categories of pages, metrics, and interactions you'd expect
from a real deployment) without touching any client's real data, name, or branding. Zones,
visitor IDs, timestamps, and happiness scores are all synthetic, generated by
[`scripts/seed.ts`](scripts/seed.ts).

## Stack

- **Next.js 16** (App Router) + **TypeScript** + **Tailwind CSS v4**
- **SQLite** via `better-sqlite3` — the mock dataset lives in `data/mock.db`, generated
  locally, never committed
- **Recharts** for bar/line/Sankey charts
- **NextAuth (Auth.js) v5** — single hardcoded/env-based credential pair gates the app
- **jsPDF** / **jspdf-autotable** for client-side PDF export

## How the mock data works

`scripts/seed.ts` generates **8 weeks of data ending "today"** (the day you run the seed).
For each day it:

1. Picks a daily visit count using a weekday/weekend multiplier (Fri/Sat busier).
2. Assigns each visit a start time from a bimodal hourly curve (morning + evening peaks).
3. Assigns a **persistent visitor ID** — about 55% of visits create a brand-new ID, the
   rest reuse an existing one from the pool, so the same anonymous passenger genuinely
   shows up across multiple visits (this matters for the At-risk page).
4. Walks the visit through one of two zone sequences (`src/lib/zones.ts`):
   - **Departure**: Check-in → Security → Immigration → (Duty Free / Food Court / VIP
     Lounge) → Departure Lounge → Gate Area
   - **Arrival**: Immigration → Baggage Claim → Arrivals Hall
5. Gives each zone stop a duration and a happiness score (1–10) with realistic noise, plus
   two low-probability modifiers: a "rough visit" (happiness knocked down across the whole
   visit) and an "extended visit" (durations stretched, simulating a delay) — these are
   what populate the At-risk page.
6. **Plants three deliberate anomalies** in the last week (a happiness drop in Security, a
   traffic spike in Duty Free, a traffic drop in Food Court) so the alerts panel always has
   something real to show, on top of whatever shows up organically.

Run `npm run seed` any time to regenerate (it's a no-op if `data/mock.db` already exists —
pass `FORCE_SEED=1` to rebuild). It's also wired as a `predev`/`prebuild` step, so a fresh
`npm install` + `npm run dev` (or a Vercel build) seeds automatically.

### Why the numbers stay consistent across pages

Every visit passes through several zones, so raw "zone footfall" (entries) is always bigger
than total visit count — a visitor who stops at 4 zones creates 4 entries. To satisfy "total
visits on Overview should equal the sum of the zone breakdown," each visit is also tagged
with a **primary zone** (wherever it spent the most time). Overview's "Visits by zone" chart
sums that field, so it always adds up to the headline total. Zone Analytics' "Traffic
density" table intentionally shows the other number — raw entries — and says so in its
subtitle, the same way the real dashboards this is modeled on distinguish "movement" from
"unique visits."

### "Today" is the dataset's last day, not your clock

Every date default (Overview's initial 30-day range, the At-risk page's "as of" date, the
forecast's starting point) is anchored to `MAX(date)` in the database, not `new Date()`.
That way the app still makes sense if you seed it once and come back to `npm run dev` weeks
later without re-seeding.

## Features

| Page | What it shows |
|---|---|
| **Overview** | KPI cards (each linking to its detail page), insight cards (busiest zone, happiest zone, zone needing attention), visits-by-zone chart (bars link to Zone analytics), gender/age donut breakdowns, forecast panel, anomaly alerts panel (each linking to Zone analytics), period-over-period comparison toggle, CSV/PDF export |
| **Visits & happiness** | Daily happiness trend, happiness by zone, day-of-week × hour heatmaps (visits volume and mood side by side), avg wait-time-by-hour chart |
| **Zone analytics** | Traffic and happiness treemaps, daily zone entries as a stacked chart (with a show/hide-weekends toggle), a traffic density table (mood + avg duration alongside volume), hour-of-day heatmap |
| **Journeys** | Sankey diagram of Entry → zones → Exit — hovering a flow highlights it through to the next zone — plus stat cards (avg zones/journey, avg dwell, departure-flow share, % exceeding a 120-min dwell) and a morning/afternoon/evening time-of-day filter |
| **At-risk passengers** | Table of anonymous, persistent visitor IDs flagged for low average happiness (≤4.5/10) or a visit with ≥120 minutes total terminal time |

Beyond the reference dashboard's scope, this demo adds:

- **Anomaly alerts** — scans the last 7 days (not just "today") and flags any zone whose
  happiness or traffic deviates ≥20% from its own trailing 14-day baseline as of that day,
  keeping the worst deviation per zone+metric (`getAnomalyAlerts` in `src/lib/queries.ts`)
- **Period-over-period comparison** — a toggle on Overview that re-runs the same KPIs over
  the immediately preceding equivalent date range and shows the % delta
- **Forecast panel** — projects the next 5 days' footfall from trailing 4-week weekday
  averages, adjusted by a recent-vs-prior 2-week momentum multiplier (a heuristic, not a
  trained model — see `getForecast`)
- **CSV/PDF export** — generated client-side from the same data already on the page, no
  server round-trip
- **Gender/age demographics** — synthetic per-visitor gender and age band, generated once per
  visitor and held constant across their visits, same as happiness and duration

## Running locally

```bash
npm install
cp .env.example .env.local   # then edit DASHBOARD_USERNAME / DASHBOARD_PASSWORD / AUTH_SECRET
npm run dev
```

Generate a real `AUTH_SECRET` with `npx auth secret` (or `openssl rand -base64 32`). The app
seeds its mock database automatically on first `npm run dev`. Sign in with whatever you set
`DASHBOARD_USERNAME` / `DASHBOARD_PASSWORD` to, then visit `http://localhost:3000`.

## Deploying

The app is stateless at runtime — `data/mock.db` is generated during the build step
(`prebuild` runs the seed script) and read-only afterwards, so it works on platforms with a
read-only filesystem like Vercel:

1. Push this repo to GitHub.
2. Import it into Vercel (or similar).
3. Set the three env vars from `.env.example` in the project settings.
4. Deploy — the build's `prebuild` step seeds a fresh 8-weeks-ending-today dataset, so the
   demo always looks current.

## Notes on scope

This is an internship demo, not a production system: there's no real camera integration,
no persistence of the auth session beyond NextAuth's JWT cookie, and the "AI-inferred mood
score" is a random number generator, not a model. Everything under `src/lib` is written to
be readable end-to-end for a stage-gate walkthrough — in particular `src/lib/queries.ts`
has one exported function per metric, each with the raw SQL inline, rather than hiding the
logic behind an ORM.
